
# План: Система бронирования мест с календарём и верификацией оплаты

## Обзор решения

Реализуем систему, где пользователи видят календарь с мастер-классами, выбирают места, а затем переходят на estafeta.ru для пожертвования. После пожертвования система проверяет появление имени и суммы на странице события.

## Архитектура

```text
+------------------+     +-------------------+     +------------------+
|   Календарь      | --> | Выбор мест        | --> | Форма оплаты     |
|   (ближайшие     |     | (20 мест,         |     | (имя, email,     |
|   мастер-классы) |     | визуальная схема) |     | кол-во мест)     |
+------------------+     +-------------------+     +------------------+
                                                           |
                                                           v
+------------------+     +-------------------+     +------------------+
|   Билет          | <-- | Верификация       | <-- | Redirect на      |
|   (генерируется  |     | (парсим estafeta, |     | estafeta.ru      |
|   после оплаты)  |     | ищем имя+сумму)   |     | для пожертв.)    |
+------------------+     +-------------------+     +------------------+
```

## Этапы реализации

### Этап 1: База данных

Создаём три таблицы:

**events** - мастер-классы:
- id (uuid)
- title (название)
- date (дата и время)
- total_seats (всего мест, например 20)
- price_per_seat (цена за место, 100₽)
- estafeta_url (ссылка на событие)
- created_at

**bookings** - бронирования:
- id (uuid)
- event_id (ссылка на событие)
- guest_name (имя гостя - для верификации)
- guest_email
- seats_count (кол-во мест)
- total_amount (итоговая сумма)
- status (pending / verified / expired)
- verification_code (уникальный код для отслеживания)
- created_at
- verified_at

**tickets** - билеты (создаются после успешной верификации):
- id (uuid)
- booking_id
- ticket_code (уникальный код билета)
- created_at

### Этап 2: Компоненты интерфейса

**EventCalendarSection** - новая секция на главной:
- Календарь в стиле референса (тёмный фон, как на chefs-table.ru)
- Подсвечиваем дни с мастер-классами
- При клике на дату показываем детали события

**SeatSelectionDialog** - модалка выбора мест:
- Визуальная схема 20 мест (4 ряда по 5 мест)
- Занятые места серые и некликабельные
- Свободные места золотые, кликабельные
- Счётчик выбранных мест и итоговая сумма

**CheckoutDialog** - форма оформления:
- Поле "Имя" (как будет на estafeta.ru)
- Поле "Email" (для отправки билета)
- Кнопка "Оплатить через Эстафету Чудес"
- Инструкция: укажите точно такое же имя при пожертвовании

**PaymentVerificationDialog** - ожидание подтверждения:
- Статус "Ожидаем подтверждение оплаты..."
- Автоматическая проверка каждые 30 секунд
- Кнопка "Я оплатил, проверить"
- Таймер: бронь действует 15 минут

**TicketDialog** - успешный билет:
- Уникальный код билета
- QR-код для входа
- Детали мероприятия

### Этап 3: Edge Functions

**verify-payment** - проверка оплаты:
1. Получает booking_id
2. Парсит страницу события на estafeta.ru
3. Ищет пожертвование с совпадающим именем и суммой
4. Если найдено - обновляет статус бронирования
5. Создаёт билет

**get-events** - список событий:
- Возвращает предстоящие мастер-классы
- Включает кол-во свободных мест

### Этап 4: Логика верификации

Парсим страницу estafeta.ru события и ищем:
- Имя донатора (указанное при бронировании)
- Сумма = кол-во мест × 100₽

Ограничения:
- Бронь активна 15 минут
- Если за это время оплата не найдена - места освобождаются
- Один человек может забронировать до 5 мест

### Этап 5: Тестовые данные

Добавляем первое событие:
- Дата: 11 апреля 2026, 11:00
- Название: "Стратегия наследия"
- 20 мест
- 100₽ за место

## Дизайн календаря

Стиль как на референсе chefs-table.ru:
- Тёмный фон (navy)
- Сетка дней недели (Пн-Вс)
- Дни с событиями подсвечены золотом
- При наведении - tooltip с названием

## Технические детали

- **RLS политики**: таблицы публичные для чтения, запись через edge functions
- **Защита**: Edge functions проверяют лимиты бронирования
- **Кэширование**: список событий кэшируется на 5 минут
- **Мобильная версия**: адаптивный календарь и модалки

## Риски и ограничения

1. **Парсинг estafeta.ru**: если структура страницы изменится, верификация перестанет работать
2. **Timing**: между оплатой и появлением на странице может быть задержка
3. **Одинаковые имена**: если два человека с одинаковым именем оплатят одновременно, может быть ложное срабатывание

## Альтернативный подход (более надёжный)

Можно добавить уникальный код в комментарий к пожертвованию на estafeta.ru, если там есть такое поле. Тогда верификация будет точнее.
